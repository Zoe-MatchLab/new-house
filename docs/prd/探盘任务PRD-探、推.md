## 1. 背景与目标

### 1.1 需求背景

**为什么要做这个需求？**

- 经纪人需要到楼盘现场采集真实物料（照片、视频、文字）
- 传统方式缺乏标准化流程，物料质量参差不齐
- 需要AI辅助生成新媒体内容，降低经纪人创作门槛
- 探盘任务目的是让经纪人在探盘时同步生成新媒体获客内容

**探盘任务的核心价值**：

1. **内容生产**：让经纪人在探盘时同步生成新媒体获客内容
2. **产品熟悉**：让经纪人真实了解楼盘，方便为客户介绍产品、解答问题

**探盘任务的内容产出**：

| 渠道 | 内容类型 |
|-----|---------|
| 小红书 | 文案、配图 |
| 公众号 | 文案、配图 |
| 朋友圈 | 文案、配图 |
| 短视频 | 脚本、视频 |

### 1.2 目标

**做完之后期望达到什么效果？**

- 标准化探盘流程，确保物料质量统一
- AI辅助内容生成，降低经纪人创作成本
- 任务进度可视化，管理层实时掌握执行情况

**如何衡量成功？**

| 指标 | 目标值 |
|-----|-------|
| 任务完成率 | ≥90%（截止前完成） |
| 物料合格率 | ≥95% |
| 内容生成效率 | AI生成≤30秒/篇 |
| 打卡位置合规率 | ≥98% |

## 4. 任务类型与状态

### 4.1 任务类型

小程序端展示两类任务，统一展示在内容日历中：

| 任务类型 | 说明 |
|---------|------|
| **下发任务** | 方便管理层移动端作业 |
| **探盘任务** | 用于经纪人执行，含3个子任务 |

### 4.2 任务状态

#### 下发任务状态（PC端）

| 状态 | 说明 | PC端可见操作 |
|------|------|-------------|
| 待下发 | 创建后未下发 | 下发、编辑、删除 |
| 进行中 | 经纪人在执行中 | 查看进度、催办、暂停 |
| 已完成 | 任务已完成 | 查看结果 |
| 已失效 | 任务已过期/暂停 | 查看记录 |

#### 探盘任务状态（小程序端）

| 状态 | 说明 | 小程序端显示 |
|------|------|-------------|
| 未开始 | 任务未开始执行 | 显示为"未开始" |
| 进行中 | 经纪人在执行中 | 显示进度 |
| 即将到期 | 任务即将到期 | 显示提醒 |
| 催办 | 被催办 | 显示催办标记 |
| 已完成 | 任务已完成 | 显示完成状态 |
| 已失效 | 任务已过期/暂停 | 显示失效状态 |

### 4.3 状态流转图

```
【下发任务】
待下发 → 进行中 → 已完成
              ↓
           已失效（超时/暂停）

【探盘任务】
未开始 → 进行中 → 已完成
              ↓
           已失效（超时）
```

## 6. 核心业务流程

### 6.1 PC端流程

```
创建任务 → 下发（下发后还可催办） → 暂停任务（暂停后不可重启） → 查看任务
```

**PC端功能说明**：
- 创建任务：选择楼盘、配置人员、配置打卡范围
- 下发：批量/单条下发，下发后给经纪人发通知
- 催办：进行中的任务可催办，**超时后不可催办**
- 暂停：暂停后任务不可重启，**删除需谨慎**
- 查看：查看任务进度、数据统计

### 6.2 小程序端探盘路书流程（3个阶段）

```
进入探盘路书任务
      ↓
【阶段1】定位获取（前置校验）
  - 后台主动获取当前位置
  - 判断是否在指定范围内
  - 不在范围内 → Toast提示用户原因，阻止进入
  - 在范围内 → 进入阶段2
      ↓
【阶段2】点位资料收集
  - 收集楼盘现场资料（照片、视频、文字描述）
  - 每次上传时校验是否在指定点位范围内
  - 不在范围内 → Toast提示需要在指定范围内上传
      ↓
【阶段3】内容生成
  - 选择需要生成的内容渠道（小红书/公众号/朋友圈/短视频）
  - 提交资料
  - 调用Coze工作流API生成内容
  - 确认内容（确认后算完成探盘路书任务）
```

**前置说明**：
- 阶段1的定位获取是进入探盘路书的前置条件
- 只有通过定位校验后才能进入点位资料收集页面
- 点位资料收集页面内也会持续校验，确保资料在指定范围内上传

### 6.3 任务完成判定

| 判定 | 说明 |
|-----|------|
| 探盘路书子任务 | 文案确认后即完成（图片/视频为耗时任务，后台继续生成） |
| 整个探盘任务 | 探盘路书 + 话术训练 + 邀约录音分析**全部完成**才算完成 |
| 无需审核 | 提交后直接进入下一状态 |

## 7. 打卡配置

### 7.1 PC端配置

| 配置项 | 说明 | 默认值 |
|-------|------|-------|
| 打卡范围 | 定位打卡范围半径（米） | 500米 |
| 打卡规则 | 先定位打卡→再点位打卡 | - |

### 7.2 小程序端执行

| 功能 | 说明 |
|-----|------|
| 定位打卡 | 到现场后校验是否在楼盘定位范围内 |
| 点位校验 | 每次点位资料上传时，校验是否在指定点位范围内；不在范围内则提醒用户需要在指定范围内上传 |

### 7.3 打卡规则

| 偏移范围 | 处理方式 |
|---------|---------|
| 200米以内 | ✅ 正常打卡通过 |
| 200-500米 | ⚠️ 弹出警告，用户确认后可继续，记录"疑似偏移" |
| 500米以上 | 🚫 强制拦截，提示"打卡位置偏离过远" |

**申诉机制**：用户可申诉 → 店长/管理员审批 → 24小时内处理

## 9. 提醒规则

### 9.1 提醒配置

由下发任务时配置，最多3个提醒时间：

| 提醒级别 | 触发时间 | 提醒方式 |
|---------|---------|---------|
| 轻度提醒 | 截止前24小时 | 推送提醒通知 |
| 中度提醒 | 截止前12小时 | 强提醒 + 短信通知 |
| 严重提醒 | 截止前2小时 | 强提醒 + 电话催办 |

### 9.2 逾期规则

- 逾期后**不可完成任务**
- 任务直接变更为"已失效"

### 9.3 超时规则

- **超时后不可催办**：任务超时后，催办按钮不可用
- **只对进行中任务催办**：只有状态为"进行中"的任务可以催办

## 12. 数据指标

### 12.1 基础指标

| 指标 | 计算方式 | 说明 |
|-----|---------|------|
| 总任务数 | SUM | 所有创建的任务数 |
| 待下发数 | COUNT(status=待下发) | 还未下发的任务数 |
| 进行中数 | COUNT(status=进行中) | 正在执行的任务数 |
| 已完成数 | COUNT(status=已完成) | 已完成的任务数 |
| 已失效数 | COUNT(status=已失效) | 已过期/暂停的任务数 |
| 完成率 | 已完成数 / 总任务数 × 100% | - |

### 12.2 分析维度

| 维度 | 指标 |
|-----|------|
| 区域维度 | 各区域任务数量、完成率、超时率 |
| 经纪人维度 | 各经纪人任务数量、完成率、平均完成时长 |
